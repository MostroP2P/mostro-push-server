Per D-07, no log lines inside this method. Designed against the post-Task-2 trait signature where `send_to_token` returns `Result<(), Box<dyn std::error::Error + Send + Sync>>`:

```rust
    pub async fn dispatch(
        &self,
        token: &RegisteredToken,
    ) -> Result<DispatchOutcome, DispatchError> {
        let mut errors: Vec<String> = Vec::new();
        let mut attempted = false;

        for (idx, service) in self.services.iter().enumerate() {
            if !service.supports_platform(&token.platform) {
                continue;
            }
            attempted = true;
            match service
                .send_to_token(&token.device_token, &token.platform)
                .await
            {
                Ok(()) => {
                    return Ok(DispatchOutcome::Delivered {
                        backend: self.backend_names[idx],
                    });
                }
                Err(e) => {
                    errors.push(e.to_string());
                }
            }
        }

        if !attempted {
            Err(DispatchError::NoBackendForPlatform)
        } else {
            Err(DispatchError::AllBackendsFailed { errors })
        }
    }
}
```

## STRIDE Threat Register

Phase 1 introduces **no new external attack surface**. The refactor is purely structural:
- No new endpoints, headers, or request paths.
- No new external dependencies (must-not #9).
- No change to `MOSTRO_PUBKEY` validation or to the Nostr filter beyond a documentation comment (the comment is informational and cannot weaken security — its purpose is to prevent a future regression toward CRIT-1).
- No change to logging shape, log levels, or sensitive-identifier truncation (D-08).
- The `Mutex<Vec<Box<dyn PushService>>>` -> `Arc<[Arc<dyn PushService>]>` change makes reads lock-free but does not loosen any trust boundary; the contained services are identical.
- Privacy posture is unchanged: same per-pubkey lookup, same per-platform dispatch, same backend HTTPS calls.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `PushDispatcher::services` post-construction | accept | The slice `Arc<[Arc<dyn PushService>]>` is immutable after `main.rs` wiring. There is no API on `PushDispatcher` to mutate it at runtime, eliminating an entire class of "service swap" attacks. No further mitigation needed. |
| T-01-02 | Information Disclosure | `PushDispatcher::dispatch` logs | mitigate | D-07 forbids any log lines inside `dispatch`; the dispatcher cannot leak `device_token`, `trade_pubkey`, or backend identifier even by accident. Verified via Task 1 acceptance grep `! grep -qE "info!|warn!|error!|debug!|trace!" src/push/dispatcher.rs`. |
| T-01-03 | Repudiation / Anti-fix regression | Future "fix" applying `.authors(mostro_pubkey)` to `Filter::new()` | mitigate | The D-11 block comment above `Filter::new()` makes the anti-requirement structurally visible at the highest-leverage location. Verified via Task 6 acceptance grep `grep -q "DO NOT add .authors" src/nostr/listener.rs &amp;&amp; grep -q "OOS-19" src/nostr/listener.rs &amp;&amp; ! grep -qE "\\.authors\\(" src/nostr/listener.rs`. |
| T-01-04 | Denial of Service | Concurrent dispatch under load | accept | The Mutex removal **reduces** DoS susceptibility (CRIT-5 in PITFALLS). The Phase 1 refactor does not introduce `tokio::spawn` or unbounded task queues (must-not #8); listener event-loop serialization remains a known pre-existing concern (CONCERNS.md "Single-task event loop blocks on each push") and is out of scope per CONTEXT.md "Untouched in Phase 1" — to be addressed in Phase 2 or a future scaling milestone. |
| T-01-05 | Elevation of Privilege | Trait surface narrowing | mitigate | D-10 deletes `send_silent_push` from the `PushService` trait. The method was unreachable in production but its presence widened the abstract attack surface (any future caller could have attempted a content-less push). Removing it shrinks the trait to only what is invoked at runtime. |

**Note on logs / privacy posture (CRIT-3 in PITFALLS):** The existing `&trade_pubkey[..16]` truncated log lines at `src/nostr/listener.rs:108, 112-116, 137` and `src/api/routes.rs:82-83, 126-130, 143-144` and `src/store/mod.rs:58-62, 70-73, 76-79` are **explicitly preserved** in this phase per D-08 / must-not #4. The privacy improvement (PRIV-01 hash-based logging + PRIV-02 `RUST_LOG=info` flip) is bundled into Phase 2 with the new `/api/notify` endpoint to keep the privacy posture changes atomic with the new external surface. Phase 1 does not regress the existing posture; it does not improve it either.
</threat_model>
